About Me

A fourth-year student at RMIT University, and aiming to be a DevOps Engineer.

I have had a strong interest in IT since I was a kid, and always tried to find ways to expand my knowledge. This would consistently show through my efforts in any given IT class throughout primary and secondary school. Heading into my second year of university, I realised that my end goal would be to work in the Cyber Security industry, as it is one of the biggest growing industries in the area of IT. This eventually changed when I did a DevOps class in 2021, which I thoroughly enjoyed. This when I realised that I would much rather enter the DevOps industry instead, which is one of the biggest IT industries in demand.

My experience with University has been amazing, as I have learnt so much over the last 3 and a half years and how you need to put in the work to get to where you want to. Admittedly, I do still struggle with concepts at times, but that's what learning is, full of ups and downs.

My present goal is to get a grasp of what it's like to work in the industry, by working in some IT entry-level jobs such as a Data Clerk. If I don't get an entry-level job by the time I graduate from University, I will be planning to do either a graduate program or an internship.

I was trying to my place in a world with ever-expanding technologies, but I am confident that I have found it in the DevOps industry. But as I continue to learn and grow, I am confident that I will make it to wherever this journey takes me.

Projects

Using CircleCI (SDO A1)

Repository

Grade: 77.5/100

Objective of assignment:
The assignment aims to understand the Continuous Integration/Continuous Deployment (CICD) methodology by focusing on the CI process. This is achieved by using CircleCI to run a CI pipeline to automate the pre-deployment process.

What was implemented:
Using a pipeline, building and testing are automated to ease manual loading. This will result in quick and easy results, whether it be a pass or fail in the pipeline. Meaning that issues can be pinpointed quicker and be fixed sooner, so changes can be moved to production.

  • Unit Testing
  • Linting
  • Code Coverage
  • Static Application Security Testing
  • Integration Testing
  • Generating a package
    • Updating workflow so the package job is only generated in the master/main branch


What was learnt:
I started to gain an understanding of the Continuous Integration process, and how it simplifies testing and deployment of solution via automating it through CircleCI. This helps in a way that it lowers manual workload, and prevents human errors such as forgetting to run a security test for example.

Using Ansible and Terraform (SDO A2)

Repository

Grade: 78/100

Objective of assignment:
The assignment aims to fully implement the CICD lifecycle by adding a CD process to the application. This is achieved by deploying an infrastructure via Terraform on AWS, and the solution is deployed onto the EC2 instance by using Ansible. Makefile commands are used here to semi-automate the process, but can be fully automated by implementing a CircleCI pipeline.

What was implemented:

  • Terraform
    • Create a VPC with 3 layers, across 3 availability zones (Public, Private and Data)
    • Create a 3 layer application infrastructure (With appropriate security groups applied to them)
      • Create a Load Balancer using the Public layer
      • Create a EC2 instance using the Private layer using the Amazon Linux image
        • The latest version was used by statically assigning it an image ID
        • Further implementation made it so it dynamically pulls the latest image ID, and uses that for the EC2 instance
      • Create a Database with DocumentDB using the Data layer
    • Create appropriate variable outputs that a shellscript can run later on
  • Ansible
    • Shellscript file
      • Create an inventory.yml file that echoes the public ip of the EC2 instance to so that Ansible can SSH into it
      • Run the Ansible-playbook command while parsing multiple terraform output variables (as env variables) that playbook.yml will use
    • Playbook and Service file
      • Create a service.tpl file
        • It contains a Unit and Service section directive
        • It will direct SystemD to where the solution is located in the EC2 instance
          • It will also automatically start the service if the instance were to reboot
        • The Database URL is dynamically created using the env variables via templating
      • Create a playbook.yml file
        • Copy solution from local to remote directory
        • Update OS and install packages to install the solutions dependencies
        • Create the service file with the tpl file and start the service
